# 3D Director Seedance-Like Motion Control Plan

## Goal

Move the 3D director motion pipeline closer to video-model-level prompt control while keeping the node deterministic, editable, and compatible with existing workflow data.

The target is not to imitate a video model internally. The target is to make the local 3D system achieve the same user-facing promise: when the user writes a motion prompt, the system can clearly understand the instruction, map it to body mechanics, generate smooth motion, and explain whether the generated result actually follows the prompt.

## Current Gap

The existing node already has the right first pieces:

- `MotionSemanticPlan`
- `resolveActionPlan`
- local action templates
- AI motion intent refinement
- `generateTransition`
- `inspectMotionQuality`
- keyframes, camera motion, recording, and preview playback

The main gap is that these pieces are still too loose:

- Prompt understanding is partly keyword based and can miss intent details.
- Some Chinese parser text still needs normalization so recognition is stable.
- Action templates describe broad motion but do not yet enforce enough real human mechanics.
- Generated motion can look like an overlay instead of a purposeful staged action.
- Quality checks can report problems but are not yet a full correction loop.
- The UI does not yet expose enough of the system's understanding for the user to inspect and correct.

## Core Principle

Use a deterministic action pipeline:

```text
Prompt
-> Semantic Plan
-> Action Stages
-> Contact / IK / Root Motion Constraints
-> Keyframe Samples
-> Quality Inspection
-> Auto Correction
-> Playable Motion Clip
```

AI is used only to improve semantic understanding. The final character motion must still be generated by the local compiler from known rig parameters, pose presets, constraints, and keyframes.

## Phase 1: Parser Normalization And Semantic Backbone

### Objective

Make the local parser reliable enough that common Chinese prompts can produce a readable and stable `MotionSemanticPlan` without AI.

### Work

- Repair remaining mojibake in motion parser, quality messages, and visible motion explanations.
- Replace scattered regex fragments with grouped dictionaries:
  - action verbs: walk, run, dash, push, pull, throw, punch, block, kick, crouch, jump, crawl, fall, get up, turn, reach
  - body focus: left hand, right hand, both hands, left foot, right foot, torso, head
  - direction: forward, backward, left, right, toward target, away from target, rotate around target
  - force and rhythm: slow, fast, sudden, heavy, light, continuous, hold
  - contact: feet on ground, knees on ground, hands touching prop, hand holding prop
  - camera: dolly in, dolly out, orbit, follow, low angle, top down, close follow, handheld
- Normalize every prompt into:
  - `actionFamily`
  - `actionType`
  - `directionLabel`
  - `speedLabel`
  - `forceLabel`
  - `bodyFocus`
  - `rootMotion`
  - `contacts`
  - `cameraIntent`
  - `targetObjectName`
  - `confidence`
  - `explain`
  - `warnings`

### Acceptance

- These prompts produce correct local semantic plans:
  - `角色双手推箱子，身体前压，双脚踩地，镜头缓慢推近`
  - `角色右手投掷石头，身体向右后方蓄力后向前甩出`
  - `角色向前跑步，镜头跟随角色`
  - `角色低重心向左侧踢`
  - `角色跳起来挥拳`
  - `角色蹲下躲避攻击`

## Phase 2: Real Human Action Stage Library

### Objective

Stop treating motion as a single overlay. Every major action should become a staged physical action.

### Work

- Add a stage template table for common actions:
  - walk: contact, transfer weight, swing leg, recover
  - run: launch, flight-like low lift, land, recover
  - dash: lean, push off, short burst, settle
  - push: prepare, contact, exert force, hold, recover
  - pull: grip/contact, lean back, pull, stabilize
  - throw: wind-up, torso twist, release, follow-through, recover
  - punch: guard, extend, impact, retract
  - block: guard raise, brace, absorb, recover
  - kick: load, lift, extend, retract, land
  - crouch: lower center, hold, optional rise
  - jump: crouch, takeoff, air, land
  - crawl: hand/leg alternation with low root
  - fall: lose balance, contact, settle
  - get up: brace, push, lift torso, stand
- Each stage must define:
  - time ratio
  - root motion tendency
  - primary body parts
  - contacts
  - allowed vertical lift
  - maximum joint amplitude
  - whether feet should remain locked
- Use pose presets as anchors when possible, but do not let presets override saved keyframes.

### Acceptance

- Generated motion no longer looks like random mid-frame distortion.
- Push, throw, punch, run, crouch, and jump have visibly different phase structures.
- Non-jump ground actions remain grounded unless the prompt explicitly asks for flying, floating, rolling, or jumping.

## Phase 3: Contact, IK, And Target Binding

### Objective

Make prompts involving props and contact points actually bind to scene objects and body targets.

### Work

- Improve target resolution:
  - If the prompt names a prop, bind to that prop.
  - If no prop name is found but the action requires a target, choose the nearest visible prop in front of the character.
  - If no valid target exists, warn clearly instead of pretending.
- For push/pull/reach/throw:
  - generate hand target constraints
  - solve hand IK toward target/contact points
  - create contact samples for reach, grasp/contact, release where applicable
- For walk/run/dash:
  - generate alternating foot contact windows
  - prevent both feet from drifting at the same time
- For crouch/kneel/crawl/fall:
  - enforce expected foot/knee/hand ground contact rules

### Acceptance

- `双手推箱子` makes both hands aim toward the box and keeps feet grounded.
- `右手投掷石头` makes right hand dominate the throw and creates a release stage.
- `向前跑步` shows alternating leg contacts and forward root movement.

## Phase 4: Trajectory Smoothing And Motion Feel

### Objective

Make the generated capture trajectory and root motion smoother and more game-like.

### Work

- Add root trajectory smoothing after sample generation:
  - position smoothing
  - yaw smoothing
  - vertical motion damping for non-airborne actions
  - endpoint preservation
- Add joint delta smoothing:
  - preserve start/end keyframes
  - smooth only inner samples
  - avoid sudden elbow/knee flips
- Add action-aware curve handling:
  - linear for mechanical steady motion
  - ease in/out for ordinary body movement
  - anticipation/release curves for throw, punch, kick
  - contact hold curves for push and pull
- Camera samples should be smoothed using the same timeline duration and keyframe times.

### Acceptance

- Root path has no obvious jitter.
- Camera orbit/follow/dolly motion no longer snaps.
- Start/end frames remain exact.
- Motion still follows prompt after smoothing.

## Phase 5: Quality Inspection And Auto Correction Loop

### Objective

Move from "generate and hope" to "generate, inspect, correct, then show result".

### Work

- Expand `inspectMotionQuality` into action-aware checks:
  - missing action type
  - missing required contact
  - non-jump action leaves ground
  - jump action has no vertical lift
  - walk/run lacks leg alternation
  - push/pull lacks hand-target contact
  - throw lacks wind-up/release/follow-through
  - excessive root displacement
  - excessive root rotation
  - large frame-to-frame joint delta
  - generated camera samples missing despite camera intent
- Add one deterministic correction pass:
  - clamp excessive amplitude
  - resmooth root trajectory
  - resmooth joint deltas
  - reapply contact locks
  - regenerate camera samples
- Store correction notes in the transition warnings or quality report.

### Acceptance

- If a generated clip violates core prompt constraints, the system either auto-fixes it or reports a specific reason.
- The UI can show "已按质量检查自动降幅/平滑/贴地" rather than silently producing bad motion.

## Phase 6: AI Refinement Contract Upgrade

### Objective

Use AI to understand language, not to invent random animation.

### Work

- Frontend request should send:
  - local semantic plan
  - available stage templates
  - current keyframes
  - character joint limits
  - scene props
  - camera list
  - current curve and duration
- Backend prompt should require:
  - structured action semantics only
  - Chinese explanation
  - no raw frame-by-frame joint rotations
  - no impossible motion unless explicitly requested
  - user explicit words override AI guesses
- Merge policy:
  - user explicit prompt > local parser > AI suggestion > defaults
  - AI may add missing details, not replace clear user instructions
- Validate AI response:
  - action type must match prompt
  - contacts must be physically plausible
  - camera intent must come from prompt or manual selection
  - force/rhythm must not exceed realistic range unless prompt asks for exaggeration

### Acceptance

- If AI returns a conflicting interpretation, local compiler keeps the user's clear intent.
- AI helps with vague prompts, but does not destabilize precise prompts.

## Phase 7: User-Facing Understanding Panel

### Objective

Let the user see and correct what the system understood before generation.

### Work

- In the dynamic panel, show a compact "系统理解结果":
  - 动作
  - 身体重点
  - 方向
  - 重心
  - 接触点
  - 阶段
  - 运镜
  - 置信度
  - 质量警告
- Keep the three-step flow:
  - `1 解析动作模板`
  - `2 AI解析动作`
  - `3 AI生成动态`
- A later optional refinement can let the user manually override misunderstood fields, but the first pass should only display and explain.

### Acceptance

- The user can tell whether the system understood the prompt before playing the result.
- If the result is wrong, the UI can explain whether the failure came from parsing, AI interpretation, target binding, or motion compilation.

## Phase 8: Browser Verification Suite

### Objective

Make this reliable by testing real prompts in the actual 3D director UI.

### Work

- Run lint after each phase.
- Browser-check these prompt cases:
  - push box with dolly-in camera
  - right-hand throw
  - forward run with follow camera
  - low center side kick
  - jump punch
  - crouch dodge
- During browser verification:
  - check no React errors
  - check generated clip exists
  - check preview plays
  - check contact/camera quality warnings
  - visually inspect whether the action matches the prompt

## Implementation Order

1. Phase 1: Parser normalization and mojibake cleanup in motion code.
2. Phase 2: Action stage template library.
3. Phase 3: Contact and target binding.
4. Phase 4: Trajectory smoothing.
5. Phase 5: Quality inspection and auto correction.
6. Phase 6: Backend AI refinement contract.
7. Phase 7: Understanding panel polish.
8. Phase 8: Full browser verification and regression cleanup.

## Non-Goals

- Do not create a second 3D director node.
- Do not split `Scene3DNode.tsx` into multiple business files during this plan.
- Do not let AI return raw animation frames.
- Do not replace saved keyframes or pose presets as user-owned truth.
- Do not modify unrelated canvas modules.

## First Implementation Target

Start with Phase 1 because current parser reliability is the root of later precision. If the text understanding layer is unstable, better IK, smoothing, or AI calls only make the wrong interpretation look smoother.
